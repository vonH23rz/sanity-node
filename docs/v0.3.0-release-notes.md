# Sanity Node v0.3.0 release notes

Release date: 2026-06-10

Sanity Node `v0.3.0` promotes the configuration-driven public runtime from a
qualified preview into a production-capable monitoring and deployment path.

This release completes the Phase 3 public-runtime migration. It adds a
supported first-run experience, fail-closed operation, production deployment
guidance, broader configuration-driven monitoring, and documented rollback
evidence.

## Highlights

- Safe workspace bootstrap and a credential-free starter configuration.
- Fail-closed validation, startup preflight, and scheduled refresh.
- Isolation between the public runtime and the original reference runtime.
- Unified Systems, Storage, Protection, and Services summaries.
- Unified public Overall Status with deterministic severity precedence.
- Atomic production generation with last-successful-output retention.
- Qualified Docker Compose installation.
- Documented advanced host-native deployment.
- Completed production cutover and rollback rehearsal.

## Monitoring scope

The configuration-driven public runtime can monitor:

- Linux and TrueNAS host system information
- host reachability and Web UI availability
- TrueNAS pool capacity and health
- disk temperatures and SMART health
- TrueNAS applications
- snapshot tasks and freshness
- replication tasks and execution state
- configured protection relationships
- collector-local and remote Linux Docker containers
- collector-local and remote Linux storage
- collector-local and remote Linux backup markers and timers
- HTTP services
- Diun and TrueNAS-native image-update state

Protection relationships can consume conservative live snapshot and
replication evidence. Image updates can annotate otherwise healthy services
without taking precedence over genuine failures.

## Installation choices

### Docker Compose

The Docker Compose path is the qualified public first-run option.

It provides deterministic bootstrap, configuration validation, startup
preflight, health checking, scheduled refresh, fail-closed invalid
configuration handling, automatic recovery after repair, and retention of the
last successful dashboard.

See [the installation guide](installation.md).

### Host-native deployment

The host-native path is intended for installations requiring collector-local
operating-system integration or durable SSH-backed remote checks.

It provides a dedicated service account, systemd-managed generation and web
serving, atomic publication, generation locking, persistent verified SSH host
trust, and explicit update, backup, rollback, and uninstall procedures.

See [the deployment guide](deployment.md).

## Fail-closed safety model

Scheduled Docker refreshes execute:

1. configuration validation
2. startup preflight
3. dashboard generation

Invalid configuration does not invoke the generator, does not activate the
reference runtime, and does not replace the last successful dashboard.

The host-native production runner also requires public mode, renders into a
temporary file, validates required output markers, and replaces production
output only after a successful complete render.

## Security

- Public mode does not invoke personal reference collectors.
- SSH host-key verification must remain enabled.
- Persistent verified `known_hosts` entries are required.
- Service-account SSH private keys must use mode `0600`.
- The host-native service uses restrictive systemd hardening.
- Output publication is atomic.

## Deployment boundaries

The unmodified Docker Compose deployment does not automatically provide:

- collector-host Docker access
- collector-host filesystem checks
- collector-host systemd inspection
- collector-local host endpoints
- persistent host-native SSH trust integration

Checks reachable from inside the container work normally. Installations that
need the capabilities above should use the advanced host-native deployment or
add an explicitly reviewed integration.

The documented host-native workspace is `/opt/sanity-node`, the default web
port is `8088`, and the generation lock is
`/opt/sanity-node/run/generator.lock`.

## Upgrade guidance

Before upgrading:

1. Back up configuration, generated output, service definitions, and SSH trust
   material.
2. Review the current example configuration and new module gates.
3. Set the intended runtime mode explicitly.
4. Validate the configuration.
5. Run startup preflight.
6. Compare or reinstall the supplied systemd units.
7. Verify service-account key ownership, mode `0600`, and persistent host
   trust.
8. Complete one successful manual generation and HTTP check before enabling
   the recurring timer.

The existing `v0.2.0` tag remains unchanged. Earlier release tags are not
moved or rewritten during the `v0.3.0` publication process.
